# 十节点流程与 UI 整合过程记录

日期：2026-07-11
基线：`origin/main` 提交 `16d3670`
集成方式：独立本地克隆与集成分支；验收前不覆盖工作目录，不推送 GitHub。

## 一、这次为什么难合并

这次问题不是普通的“同一行代码冲突”，而是功能组和 UI 组同时修改了同一批高耦合资产：

1. **双方同时修改 Unity 场景。** `.unity` 文件本质是引用大量对象、组件、GUID 和 fileID 的 YAML。两边分别新建或重排对象后，即使 Git 能自动合并文本，也不能证明按钮引用、脚本引用和层级关系仍然正确。
2. **UI 不是基于统一场景继续搭建。** UI 版重新建立了部分场景和对象结构，导致旧脚本期待的对象名、按钮事件、场景名和新 UI 实际结构不一致。
3. **跨场景状态没有唯一负责人。** 地图、战斗、选牌和事件各自修改 `isBattleWin`、`DraftMode`、节点解锁等字段，出现了事件面板一进地图就显示、事件节点点不进去、战斗胜利跳过奖励、旧节点可以重打等问题。
4. **“保存当前状态”和“保存安全点”语义不同。** 如果战斗中点击退出时直接序列化当前 `GameManager`，就会把战中扣血写进存档，破坏“继续后回到战前地图”的设计。
5. **终局规则发生了变化。** 原代码按三层、每层十节点设计，并在最终层后重置到第一层；现在需求是一局只有十节点，旧的推进函数会让通关后重新开始。
6. **卡牌升级与重复牌需要保存具体实例。** 永久牌组虽然仍是字符串列表，但升级牌用 `cardId+` 表示。删牌或升级若按 `cardId` 去重或使用 `Remove(cardId)`，会删错重复牌或漏掉升级牌。
7. **字体与二进制资源不适合并行改。** TMP 动态字体图集会在编辑器运行后产生脏修改；图片、场景、字体等资源又无法像普通 C# 一样做逐行语义合并。

## 二、本次采用的合并办法

### 1. 先保护现场，再同步基线

- 单独备份本地动态字体资产。
- 确认本机工作区没有其他未保存改动。
- 只使用 `git merge --ff-only origin/main` 快进到最新主线。
- 从同一提交建立独立集成克隆和 `integration/full-loop-20260711` 分支。

这样可以把“同步主线”“整合功能”“安装回工作目录”分成三个独立步骤，任何一步出错都不会破坏开发者原工作区。

### 2. 先审计状态流，再改 UI

审计先确认了以下真实故障：

- `EventChoicePanel` 默认激活，但事件节点点击逻辑并未打开它。
- 战斗场景的 `_returnToMapOnWin` 为 false，导致设置 `BattleReward` 的代码永远不执行。
- 地图入口只检查 unlocked，不检查 cleared，所以通关节点可重复进入。
- 第十节点仍调用三层推进逻辑，会再次解锁第一节点。
- 排行榜场景只有背景，没有读取 SQLite 并显示记录的控制器。
- “继续游戏”按钮没有任何回调。

在这些状态问题没有明确前，直接继续调场景只会制造更多表面正常、实际断流的 UI。

### 3. 把流程收敛为一个十节点状态机

固定节点表：

| 节点 | 类型 |
| --- | --- |
| 1–2 | 战斗 |
| 3 | 事件 |
| 4 | 休息 |
| 5–7 | 战斗 |
| 8 | 事件 |
| 9 | 休息 |
| 10 | 普通战斗（临时） |

地图点击时同时检查：

- 节点已解锁；
- 节点未通关；
- 节点正好是第一个未通关节点。

所有节点完成后统一经过 `RunFlowCoordinator`：标记完成、解锁下一节点、记录最高进度、清理场景瞬态、保存安全点、回地图。第十节点胜利和任意节点战败走独立终局入口，不再推进到新层。

### 4. 把“安全点存档”和“只更新时间”拆成两个 API

- `RunSaveRepository.SaveSafePoint()`：只在合法地图安全状态写入 HP、牌组、元素、节点进度和挑战计时快照。
- `RunSaveRepository.UpdateChallengeStateOnly()`：战斗中退出、应用失焦或关闭时，只更新计时字段，绝不覆盖战前地图状态。

继续游戏恢复后还会强制清空格挡、能量、手牌、抽牌堆、弃牌堆、战斗结果和临时选牌模式，保证不会伪造“半场战斗恢复”。

### 5. UI 通过公共控制器接入，不再复制业务代码

- `EventSceneController` 只负责“获得卡牌/移除卡牌”选择。
- `RestSceneController` 只负责“回复 15 点/升级卡牌”选择。
- `CardDraftManager` 根据模式显示奖励、删牌或升级，并按 `playerCardBag` 的实际索引处理重复牌。
- `RunHudController` 为非主菜单场景统一提供 HP、牌组、退出确认，以及战斗抽牌堆/弃牌堆查看。
- 所有卡牌查看都调用现有 `CardView`，不再维护另一套容易不同步的卡面。

### 6. 场景接线由一次性 Editor 脚本完成

新增的 Editor 集成脚本负责：

- 生成 `EventScene` 和 `RestScene`；
- 加入 Build Settings；
- 保证 `MainMenuScene` 位于构建索引 0；
- 给主菜单和排行榜挂控制器；
- 关闭地图旧事件面板；
- 写入固定节点类型。

运行时代码不依赖手工拖拽大量引用，减少后续再次出现“某个人场景里有、另一个人场景里没有”的情况。

## 三、后续协作规则

### 代码所有权

- 功能组负责：`GameManager`、`RunFlowCoordinator`、存档结构、战斗/节点结算规则。
- UI 组负责：场景视觉、布局、图片、动画和预制体。
- 集成人负责：场景入口、按钮到公共 API 的接线、Build Settings 和全流程验收。

任何一组需要修改另一组负责的文件时，先提交接口需求，不要直接复制或重写对方逻辑。

### 场景规则

1. 不再为同一流程私自新建第二套同名用途场景。
2. 业务脚本不得通过层级路径跨场景寻找对象。
3. UI 按钮只调用稳定的公共入口，不在按钮回调里重复实现结算。
4. 公共 HUD、弹窗、卡牌查看优先做成复用组件或运行时控制器。
5. Unity 自动生成的 `.meta` 必须和对应资源一起提交。

### Git 工作流

1. 开始任务前从最新 `main` 建短分支。
2. 场景和业务代码尽量分成不同提交。
3. 同一场景同一时间只指定一个负责人修改。
4. 合入前先更新主线，再由场景负责人解决一次场景差异。
5. 合并成功不等于功能成功；必须跑 Unity 编译和完整场景闭环。

## 四、必须执行的验收矩阵

1. 新游戏删除旧临时存档，从点击开始时计时。
2. 开局选牌后地图只解锁节点 1。
3. 1–10 节点类型与固定表一致。
4. 已完成节点在 UI 和入口层都不能再次进入。
5. 战斗中退出后继续：回地图、恢复战前 HP/牌组、当前节点仍可重试。
6. 战斗胜利领取或跳过奖励后，才完成节点并保存。
7. 事件获得卡牌使用战斗奖励池；删牌能区分重复牌和升级牌。
8. 休息回复 15 点且不超过上限；升级后保存为 `cardId+`，战斗显示升级效果。
9. 所有非主菜单场景显示 HP、牌组和退出确认；战斗可查看实时抽/弃牌堆。
10. 退出期间、应用失焦期间不计入实际游玩时长。
11. 战败和第十关胜利各只写一条排行榜记录，并删除继续存档。
12. 第十关结束后不能再次解锁第一关。
13. 排行榜显示开始时间、实际用时、最高完成进度和结果。
14. Unity 编译无错误，所有新增场景已进入 Build Settings，构建入口为主菜单。

只有上述矩阵全部通过，才能把集成提交安装回正式工作目录。

## 五、最终排障与验收结论

### 1. 删牌 UI 的真实回归

第一轮 Play Mode 测试发现，`BuildOwnedOptions()` 已经按牌组索引生成候选，但
`BuildOwnedSelectionUi()` 开头调用的清理函数又清空了 `_ownedOptions`，所以界面显示
0 张可删牌。修复方式是把“销毁旧 UI”和“清空候选数据”拆成两个生命周期；新的候选
只在新一轮构建前和本轮结束后清理。这个问题说明：对象存在、场景能跳转并不能证明
数据绑定正确，必须实际点击并检查生成的卡面数量。

### 2. 排行榜 SQLite 兼容问题与最终选择

原项目的 `Assets/Plugins/Mono.Data.Sqlite.dll` 来自 Unity 的 `lib/mono/unity` 桩目录，
`SqliteConnection.CreateCommand()` 在编辑器运行时只有非法 IL。替换成 Windows JIT
托管实现并加入原生 SQLite 后虽然能运行到终局写入，但在 Unity 2022.3.62f3 发生了
原生 `SIGSEGV`。继续绑定不同平台的托管/原生二进制会把排行榜变成高风险部署点。

最终保留 `ChallengeRecordRepository` 的公共接口和排序规则，把内部存储改为独立的
`challenge_records.json`：

- 与 `run_save.json` 仍是两个互不覆盖的文件；
- 使用临时文件、`File.Replace` 和备份文件完成原子提交；
- 文件损坏时拒绝覆盖旧记录；
- 排序仍为最高进度降序、有效用时升序、结束时间降序；
- 不再依赖平台原生 DLL，可在编辑器和玩家构建中保持一致。

### 3. 保存失败不再静默跳转

- 进入节点前若安全点写入失败，会恢复点击前的 `currentNode` 等字段并取消切场景。
- 节点结算安全点最多重试三次，仍失败时保留当前场景，不用未落盘状态继续前进。
- 齿轮退出或地图返回主菜单时，计时快照写入失败会取消退出并给出提示。
- 齿轮确认框打开时暂停主动计时，点击“否”才恢复计时。
